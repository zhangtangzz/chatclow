RAG 知识库检索 · Tool/Function Calling 工具调用 · 完整数据流

📌 整体架构一览

对话请求经过责任链 7 个 Step。其中与数据库相关的两大链路：

ContextAssemblyStep → RagStep → MessageBuildStep → ModelCallStep / SyncAiCallStep → FunctionCallingStep → RecordSaveStep → ResponseFinalizeStep
RAG 链路 RagStep(@Order=20) — 从知识库检索相关内容拼入上下文
Tool 链路 MessageBuildStep(@Order=25) + FunctionCallingStep(@Order=40) — 加载工具定义 → AI 选择调用 → 本地执行

二、表结构 & 字段映射

① chatclow_ai_agent → chatclow_rag_kb

ai_agent.kb_id ══════ → ══════ rag_kb.id（外键关联）
Agent 字段类型说明
kb_enabledINT (0/1)是否启用 RAG。=0 时 RagStep.shouldSkip() 返回 true，跳过检索
kb_idBIGINT绑定的知识库 ID，指向 rag_kb.id。为 null 时 also skip
rag_top_kINT (默认6)双路检索取多少条候选项
rag_final_kINT (默认3)Rerank 后最终保留几条送 LLM
rag_similarity_thresholdFLOAT (默认0.5)向量相似度阈值，低于此值的切片过滤掉
rag_keyword_enabledINT (默认1)是否启用关键字检索（双路检索）
rag_rerank_enabledINT (默认1)是否启用 Rerank 重排序

② chatclow_rag_kb → chatclow_store_instance

rag_kb.store_instance_id ══════ → ══════ store_instance.id
rag_kb 字段类型说明
store_instance_idBIGINT向量存储实例 ID，决定 chunk 存到哪个后端（MySQL/MongoDB/pgvector）
vector_store_typeVARCHAR冗余字段，方便查询（如 MYSQL、MONGODB、PGVECTOR）
embedding_model_idBIGINT绑定的 Embedding 模型 ID（用于文本向量化）
user_idBIGINT创建者 ID。用户只能管理自己的 KB，管理员可管理全部
statusINT (0/1)知识库状态
store_instance 字段类型说明
idBIGINT主键，被 rag_kb.store_instance_id 引用
nameVARCHAR实例名称（如 "MySQL-默认实例"）
store_typeVARCHAR存储类型：MYSQL / MONGODB / PGVECTOR
config_jsonTEXTJSON 连接配置

③ chatclow_rag_kb → chatclow_rag_document → chatclow_rag_chunk

rag_kb.id ══════ → ══════ rag_document.kb_id（知识库下的文档）
rag_document.id ══════ → ══════ rag_chunk.document_id（文档下的切片）
rag_kb.id ══════ → ══════ rag_chunk.kb_id（冗余字段，方便按知识库检索）

三、shouldSkip 逻辑（决定 RAG 是否执行）

// RagStep.shouldSkip()
return agent == null
    || agent.getKbEnabled() == null
    || agent.getKbEnabled() != 1    // kb_enabled != 1 → 跳过
    || agent.getKbId() == null;     // kb_id == null → 跳过

通过 = 三个条件全部不满足 → 执行 RAG 检索
跳过 = 任意一个满足 → 直接跳过，不检索

四、RAG 检索流程

用户问题(query)
    │
    ├──① EmbeddingService.embed(query) ──────────────→ float[] 向量
    │
    ├──② VectorStoreFactory.getVectorStore(kbId) ────→ 根据 kbId 找到 rag_kb
    │                                                    → 读 store_instance_id
    │                                                    → 创建对应后端的 VectorStore
    │
    ├──③ 向量检索：vectorStore.search(vec, kbId, topK, threshold)
    │      → 在 chunk 中找最相似的 topK 条
    │
    ├──④ 关键字检索（如启用）：vectorStore.keywordSearch(query, kbId, topK)
    │      → 用 LIKE 或全文索引检索
    │
    ├──⑤ 合并去重（按 chunk.id）
    │
    └──⑥ Rerank（如启用）→ 最终取 ragFinalK 条
         → 拼接到 ctx.ragContext → MessageBuildStep 拼入 system prompt

二、表结构 & 绑定方式

① 数据库定义：chatclow_ai_function

字段类型说明
idBIGINT (PK)主键
agent_idBIGINT外键 → ai_agent.id，决定哪个智能体拥有哪些工具
nameVARCHAR(100)工具名（如 get_weather）— 用于匹配执行器
descriptionVARCHAR(500)工具描述，AI 据此类判断何时调用
parametersTEXT参数定义（JSON Schema 格式），告诉 AI 需要哪些参数
statusINT状态

② 加载时机：MessageBuildStep

// MessageBuildStep 中加载工具定义
List<AiFunction> functions = aiFunctionService.listByAgentId(agentId);
// → SELECT * FROM chatclow_ai_function WHERE agent_id = ? AND status = 1

// 拼装成 OpenAI tools JSON 传给 AI
// AI 根据 description + parameters 决定是否调用哪个工具
// AI 返回 tool_calls → FunctionCallingStep 执行

③ 执行匹配：FunctionExecutorImpl（硬编码注册表）

// FunctionExecutorImpl.init() — 写死在代码里，与数据库无关
registry.put("get_weather",      this::getWeather);       // 查天气
registry.put("get_current_time", this::getCurrentTime);   // 当前时间
registry.put("calculate",        this::calculate);        // 四则运算

// execute() 方法：
String result = registry.get(functionName).apply(args);
// 如果 functionName 不在 registry 中 → 返回 "未知工具"
⚠️ 关键限制：数据库中的 ai_function.name 必须与 registry 中的 key 完全一致（字符串匹配）。数据库只存定义（description + parameters），实际执行逻辑全部在 Java 代码中。

⏱️ 完整请求数据流

以"代码助手"为例，一次对话请求的数据库访问时序：

用户发送消息 "Java 的 List 和 Set 有什么区别？今天天气怎么样？"
    │
    │  【ContextAssemblyStep】
    ├─→ SELECT * FROM chatclow_ai_agent WHERE id = ?       ← 加载 Agent 配置
    ├─→ SELECT * FROM chatclow_ai_model WHERE id = ?       ← 加载模型配置（apiKey 自动解密）
    ├─→ INSERT INTO chatclow_ai_conversation ...            ← 新建会话（首次）
    ├─→ SELECT * FROM chatclow_ai_conversation_record ...  ← 加载历史消息（如果 memoryEnabled）
    │
    │  【RagStep】（kb_enabled=1, kb_id=1 → 不跳过）
    ├─→ rag_kb → store_instance → 确定向量存储后端        ← 读配置
    ├─→ Embedding API 调用（SiliconFlow）                   ← 向量化
    ├─→ vectorStore.search() → 查 rag_chunk 表             ← 向量检索
    ├─→ Rerank API 调用（SiliconFlow）                      ← 重排序
    │
    │  【MessageBuildStep】
    ├─→ SELECT * FROM chatclow_ai_function WHERE agent_id=? ← 加载工具定义
    ├─→ 拼接 system prompt（含 RAG 上下文）
    ├─→ 拼接 tools JSON（含 3 个工具定义）
    │
    │  【ModelCallStep / FunctionCallingStep】（调用 AI API）
    ├─→ AI 返回：回答 List/Set 区别 + 调用 get_weather()
    ├─→ FunctionExecutorImpl.execute("get_weather", {"city":"北京"})
    │   → 返回 {"weather":"晴","temperature":25}
    ├─→ 第二次 AI 调用（带工具结果）
    │
    │  【RecordSaveStep】
    └─→ INSERT INTO chatclow_ai_conversation_record ...    ← 保存对话记录
    └─→ UPDATE chatclow_ai_conversation SET total_tokens=? ← 更新 token 累计
    │
    ⚠ 异常时：发送 SSE error 事件 → 前端显示重试横幅（见下）

💡 关键总结

RAG 链路要点

  • 入口开关：ai_agent.kb_enabled=1 + kb_id != null
  • 中间跳转：rag_kb.store_instance_id → store_instance.id → 决定向量存储后端
  • 实际数据：rag_document 和 rag_chunk 通过 kb_id 关联
  • 当前只有"代码助手" 能真正使用 RAG
  • "讲笑话agent"的 kb_id=3 引用了不存在的知识库，是脏数据
  • 级联删除已修复：删除知识库时自动清理文档、切片和向量存储数据，不会再产生孤立脏数据

Tool 链路要点

  • 绑定方式：ai_function.agent_id → ai_agent.id
  • DB 只管定义：name / description / parameters（JSON Schema）
  • 代码管执行：FunctionExecutorImpl.registry 硬编码 3 个工具
  • 名字必须一致：DB 的 name = registry 的 key（字符串匹配）
  • 当前只有"代码助手" 绑定了 3 个工具
  • 给其他 Agent 加工具：只需在 ai_function 表插入 agent_id=X 的记录，name 用已有的即可
  • 级联删除已修复：删除智能体时自动清理关联的工具定义记录
📌 一句话总结：
RAG 能用的前提是 Agent 设置了 kb_enabled=1 + kb_id 指向一个真正存在且有切片的知识库。
Tool 能用的前提是 ai_function 表里有该 Agent 的记录，且 name 与代码中的 registry key 一致。
当前只有 "代码助手" 同时满足这两个条件。
