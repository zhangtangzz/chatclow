## 一、今日完成内容

### 1. 多轮对话核心功能实现

**问题背景：**
之前每次发消息 AI 都"失忆"，因为每次请求都是独立的新对话，AI 看不到历史消息。

**解决方案：**
LLM API 的 `messages` 数组支持传入历史消息，实现多轮上下文。

**代码改动：**

#### ① ChatService 接口改造
```java
// 之前：只返回 String（AI 回复文本）
String chat(Long agentId, Long userId, String message);

// 之后：返回 ChatResponse（含 conversationId）
ChatResponse chat(Long agentId, Long userId, String message, Long conversationId);
```

#### ② ChatServiceImpl.chat() 核心逻辑
```java
@Override
public ChatResponse chat(Long agentId, Long userId, String message, Long conversationId) {
    // 1. 查智能体配置
    AiAgent agent = aiAgentMapper.selectById(agentId);
    
    // 2. 查模型配置
    AiModel model = aiModelMapper.selectById(agent.getModelId());
    
    // 3. 获取或复用会话（核心！）
    AgentConversation conversation;
    if (conversationId != null) {
        conversation = conversationMapper.selectById(conversationId); // 复用
    } else {
        conversation = createNewConversation(userId); // 新建
    }
    
    // 4. 保存用户消息
    saveRecord(conversation.getId(), "user", message);
    
    // 5. 查询历史消息，拼入 messages 数组
    LambdaQueryWrapper<AgentConversationRecord> wrapper = new LambdaQueryWrapper<>();
    wrapper.eq(AgentConversationRecord::getConversationId, conversation.getId());
    wrapper.orderByAsc(AgentConversationRecord::getCreatedDt);
    List<AgentConversationRecord> history = recordMapper.selectList(wrapper);
    
    // 6. 调用 AI API（带上历史上下文）
    String aiReply = callAiApi(model, agent.getSystemPrompt(), history, message);
    
    // 7. 保存 AI 回复
    saveRecord(conversation.getId(), "assistant", aiReply);
    
    // 8. 返回 AI 回复 + 会话 ID
    return new ChatResponse(aiReply, conversation.getId());
}
```

#### ③ callAiApi() 改造 — 拼装 messages 数组
```java
private String callAiApi(AiModel model, String systemPrompt,
                         List<AgentConversationRecord> history, String userMessage) {
    // ... 构建请求 ...
    
    List<Map<String, String>> messages = new ArrayList<>();
    
    // ① 先放 system 提示词
    messages.add(Map.of("role", "system", "content", systemPrompt));
    
    // ② 再放历史消息（user + assistant 交替）
    for (AgentConversationRecord record : history) {
        messages.add(Map.of("role", record.getRole(), "content", record.getContent()));
    }
    
    // ③ 最后放当前用户的新消息
    messages.add(Map.of("role", "user", "content", userMessage));
    
    requestBody.put("messages", messages);
    // ... 发送请求，解析响应 ...
}
```

### 2. 返回值优化 — ChatResponse DTO

**问题：** 之前 `/api/chat/send` 只返回 AI 回复文本，前端不知道 `conversationId`，无法续接对话。

**解决：** 新建 `ChatResponse` DTO，同时返回回复内容和会话 ID。

```java
// dto/ChatResponse.java
public class ChatResponse {
    private String reply;           // AI 回复内容
    private Long conversationId;     // 会话 ID（前端续接用）
    
    public ChatResponse(String reply, Long conversationId) {
        this.reply = reply;
        this.conversationId = conversationId;
    }
    // getter/setter ...
}
```

**前端拿到的数据：**
```json
{
  "code": 200,
  "data": {
    "reply": "你刚才告诉我你叫张三呀！",
    "conversationId": 1
  }
}
```

### 3. DTO 包重构 — 规范分层

**问题：** `ChatRequest` 写在 `ChatController` 内部类里，`ChatResponse` 放在 `entity/` 包不合理（它不是数据库实体）。

**解决：** 新建 `com.chatclow.dto` 包，把请求/响应对象移过去。

```
com.chatclow/
├── entity/    ← 数据库实体（User、AiAgent、AgentConversation...）
├── dto/       ← 数据传输对象（ChatRequest、ChatResponse）← 新增
├── mapper/
├── service/
├── controller/
└── ...
```

**Entity vs DTO 区别（面试必考）：**

| | Entity | DTO/VO |
|---|---|---|
| 对应 | 数据库表 | 接口返回/接收 |
| 包含 | 表里所有字段 | 只给前端需要的 |
| 敏感信息 | 可能有密码等 | 必须过滤掉 |
| 例子 | `User`（含 password） | `UserVO`（不含 password） |

## 三、遇到的问题及修复

### 问题 1：LambdaQueryWrapper 用法错误

**报错：**
```
java: 找不到符号
  符号:   方法 lambdaQuery()
  位置: 类型为 com.chatclow.mapper.AgentConversationRecordMapper 的变量 recordMapper
```

**原因：** `lambdaQuery()` 是 `ServiceImpl` 的快捷方法，普通 `Mapper` 没有这个方法。

**修复：** 改用 `new LambdaQueryWrapper<>()` 手动构建：
```java
// ❌ 错误
List<AgentConversationRecord> history = recordMapper.selectList(
    recordMapper.lambdaQuery()...getWrapper()
);

// ✅ 正确
LambdaQueryWrapper<AgentConversationRecord> wrapper = new LambdaQueryWrapper<>();
wrapper.eq(AgentConversationRecord::getConversationId, conversation.getId());
wrapper.orderByAsc(AgentConversationRecord::getCreatedDt);
List<AgentConversationRecord> history = recordMapper.selectList(wrapper);
```

### 问题 2：ChatService 返回类型不匹配

**报错：**
```
不兼容的类型: java.lang.String 无法转换为 com.chatclow.dto.ChatResponse
```

**原因：** `ChatController` 已改为期望 `ChatResponse`，但 `ChatService` 接口还返回 `String`。

**修复：** `ChatService.chat()` 返回类型改为 `ChatResponse`：

## 四、学习要点总结

### 1. 多轮对话核心原理

```
用户提问："我叫什么名字？"
    ↓
ChatService.chat(conversationId=1)
    ↓
① 查数据库：会话 1 的历史消息
    [user: "你好", assistant: "...", user: "我叫张三", assistant: "..."]
    ↓
② 拼装 messages 数组：
    [
      {role: "system", content: "你是助手"},
      {role: "user", content: "你好"},
      {role: "assistant", content: "..."},
      {role: "user", content: "我叫张三"},
      {role: "assistant", content: "..."},
      {role: "user", content: "我叫什么名字？"}  ← 当前问题
    ]
    ↓
③ 调用 AI API → AI 看到完整历史 → 回答："你叫张三"
```

**关键点：**
- `conversationId` 要从返回值传给下一次请求
- 历史消息按时间升序排列
- `system` 消息永远在最前面

---

### 2. Entity vs DTO 分层设计

**为什么要分层？**
- **安全性：** Entity 可能含密码等敏感字段，不能直接返回前端
- **解耦：** 数据库表结构变化不影响接口格式
- **灵活性：** 同一个 Entity 可以对应多个 DTO（列表页 DTO、详情页 DTO）

**命名规范：**
- 请求对象：`XxxRequest` 或 `XxxDTO`
- 响应对象：`XxxResponse` 或 `XxxVO`
- 有的团队全叫 DTO，有的分 Request/Response，看公司规范

---

### 3. 项目包结构规范（企业级）

```
com.chatclow/
├── ChatclowApplication.java     ← 启动类
├── common/                    ← 公共类（R 统一响应）
├── config/                    ← 配置类（WebMvcConfig）
├── exception/                 ← 全局异常处理
├── interceptor/              ← 拦截器（JwtInterceptor）
├── util/                     ← 工具类（JwtUtil）
├── entity/                   ← 数据库实体（@TableName）
├── dto/                      ← 数据传输对象（Request/Response）
├── mapper/                   ← MyBatis Mapper 接口
├── service/                  ← 业务接口
│   └── impl/                ← 业务实现
└── controller/               ← REST 接口层
```
